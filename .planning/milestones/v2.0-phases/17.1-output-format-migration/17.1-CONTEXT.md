# Phase 17: Output Format Migration & Context Depth Fix

## Objective
Migrate output format to use `<file name="package.class">` tags and fix `get_context` depth logic to be more precise and functional.

## Decisions
1.  **Output Format:**
    -   Replace class-specific tags (e.g., `<UserController>`) with a standard `<file name="package.Class">` tag.
    -   Example: `<file name="com.cre.mcp.UserController"> ... </file>`.
2.  **Depth Logic:**
    -   **`depth = 1`**: Return only the target function code, wrapped in its `<file>` tag. All other class members (fields, other methods) must be omitted.
    -   **`depth > 1`**: Include the target function AND the specific callees reached by the traversal (at the specified depth).
    -   **Strict Reachability**: Omit all functions and properties that are NOT part of the reachability set for the given depth.
3.  **Edge Filtering**:
    -   `gatherNodesOrdered` must be selective about which edges it follows. For "Context Reconstruction", it should primarily follow `CALLS` and `USES_FIELD`.
    -   Avoid jumping from Method to Class via `BELONGS_TO` and then to all other methods via `ENTRY_POINT`/`SERVICE_LAYER` unless the user explicitly requested the Class context.
4.  **Bug Fixes**:
    -   Address why `depth > 1` currently returns all functions: This is likely due to the BFS following `ENTRY_POINT` edges from the parent Class node.
    -   Address why `depth > 2` fails to retrieve deeper functions: This is likely due to failed symbol resolution in `JavaAstIndexer.resolveCallee` (especially for complex arguments).
5.  **Verification**:
    -   A Python test script will be created at `scripts/test_context_depth.py` to verify directly against `/home/hainn/blue/code/cre-test-project`.

## Strategy
-   **Modify `CreServiceImpl.buildIntegratedView`**:
    -   Use `typeFqn` for the `<file name="...">` tag.
    -   Remove the logic that generates `<ClassName>` tags.
-   **Refactor `CreServiceImpl.gatherNodesOrdered`**:
    -   Add edge type filtering to prioritize traversal through `CALLS` and `USES_FIELD`.
    -   Ensure `depth=1` returns ONLY the start node.
-   **Refactor `CreServiceImpl.transformWithRelevance`**:
    -   Ensure it strictly honors the `retainedNodes` set for ALL members (methods and fields).
    -   If a member is not in `retainedNodes`, it MUST be pruned, regardless of other relevance heuristics.
-   **Improve `JavaAstIndexer`**:
    -   Enhance `resolveCallee` to handle more complex argument types and chained calls if necessary to ensure deep traversal is possible.
