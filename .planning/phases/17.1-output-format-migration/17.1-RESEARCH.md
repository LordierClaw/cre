# Research: Phase 17 - Output Format Migration & Context Depth Fix

## Empirical Observations

### 1. "All Functions" Bug at Depth > 0
- **Finding**: When `depth > 0`, the BFS in `gatherNodesOrdered` follows `BELONGS_TO` from a Method to its Class, and then potentially `ENTRY_POINT` or `SERVICE_LAYER` edges from the Class back to ALL its methods.
- **Evidence**: 
  - `AdminBookController::createBook` belongs to `AdminBookController`.
  - `AdminBookController` has `ENTRY_POINT` edges to `getBook`, `getAllBooks`, etc.
  - If `depth >= 2` (or even 1 if starting at Class), all these methods are gathered.
- **Root Cause**: `gatherNodesOrdered` does not filter edge types. It follows everything except `CATCH_INVOKES`.

### 2. Output Format Migration
- **Current State**: Uses `<ClassName>` tags generated by replacing the class code in `buildIntegratedView`.
- **Target State**: `<file name="package.Class">` replacing the class tag.

### 3. Depth > 2 Failure
- **Finding**: While `depth=2` worked for `AdminBookController -> BookService -> BookRepository`, deeper chains might fail due to `JavaAstIndexer`'s simple signature matching.
- **Observation**: `BookRepository::save(Book)` might be resolved as `BookRepository::save(?)` if the argument type isn't perfectly inferred, causing a mismatch and stopping the BFS.

## Proposed Fixes

### 1. Strict BFS Traversal
- Modify `gatherNodesOrdered` to ONLY follow `CALLS` and `USES_FIELD` when traversing for context.
- (Optional) Follow `IMPLEMENTS` to reach implementation methods if the start node is an interface method.

### 2. Precise Pruning
- Refactor `transformWithRelevance` to strictly remove any `MethodDeclaration` or `ConstructorDeclaration` whose `nodeId` is not in the `gathered` list.
- Ensure `depth=0` (which results in `gathered = [startNode]`) properly prunes everything else.

### 3. XML Format Update
- Update `buildIntegratedView` to wrap the entire `CompilationUnit` output (or just the primary type) in `<file name="...">`.
- Remove the `<ClassName>` tag logic.

### 4. Indexer Robustness
- Improve `JavaAstIndexer.resolveCallee` to better handle method signatures, especially for standard library or third-party types where exact matching is hard.
